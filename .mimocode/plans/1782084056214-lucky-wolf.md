# Plano: Prontuário Eletrônico Completo — HOVET.2

## Contexto

O HOVET.2 é um sistema de gerenciamento de filas para hospitais veterinários universitários. Atualmente gerencia apenas a fila de pacientes (senha → triagem → atendimento → finalização) sem nenhuma informação de prontuário, tutores ou histórico médico. O objetivo é adicionar um **Prontuário Eletrônico Completo** com: cadastro de tutores/pacientes, registros de consulta, histórico médico e agendamento.

## 4. Engine — Operações CRUD

**`src/core/engine.ts`** — Adicionar funções (ou criar `src/core/ehr.ts` para separação):

- `searchOwners(query, unidade)` — busca por nome/CPF/telefone

- `createOwner(data)` — criar tutor

### Patients

### Consultas

- `createConsulta(data)` — nova consulta

### Vacinas
- `getVacinasByPatient(patientId)` — histórico de vacinas
- `createVacina(data)` — registrar vacina
- `deleteVacina(id)` — deletar

### Cirurgias
- `getCirurgiasByPatient(patientId)` — histórico

### Exames
- `getExamesByPatient(patientId)` — histórico
- `createExame(data)` — registrar exame

## 7. SQL — Políticas RLS

Para cada nova tabela:
- `INSERT`: Qualquer authenticated
- `SELECT`: Admin vê tudo, demais vêem da própria unidade
- `UPDATE`: Admin ou owner do registro
- `DELETE`: Admin only

## 10. Decisões de Design

- **Engine separado**: Criar `src/core/ehr.ts` em vez de poluir `engine.ts` (que já tem 537 linhas)
- **IDs únicos**: Manter padrão `prefix-timestamp-random` existente
- **snake_case no DB / camelCase no TS**: Seguir convenção existente
- **Multi-campus**: Todas as tabelas têm coluna `unidade` + RLS por unidade
- **Vinculação opcional**: `pets.patient_id` é nullable — fila continua funcionando sem prontuário
- **Formulários modais**: Usar `createPortal` como o projeto já faz (ver `Recepcao.tsx` e `PetListRow.tsx`)
